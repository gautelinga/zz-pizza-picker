Add the following date-formatting block to the `index.astro` frontmatter immediately after
the existing `const menu = menuData as MenuData;` line (per D-03):

```typescript
const scrapedAt = new Date(menu.scraped_at);
const day = scrapedAt.getUTCDate();                          // no leading zero (7, not 07)
const month = scrapedAt.toLocaleString('en-GB', { month: 'short', timeZone: 'UTC' });
const year = scrapedAt.getUTCFullYear();
const lastUpdated = `Menu last updated: ${day} ${month} ${year}`;
```

CRITICAL: Use `getUTCDate()` and `getUTCFullYear()` (NOT `getDate()` / `getFullYear()`), and
pass `timeZone: 'UTC'` to `toLocaleString`. This prevents ±1 day drift when the CF build
machine's local timezone differs from UTC (Pitfall 4 in RESEARCH.md).

```yaml
# .github/workflows/weekly-rebuild.yml
# Triggers a Cloudflare Pages rebuild on a weekly schedule (DEPL-02).
# CF Pages does the actual build — this job only fires the deploy hook.
# Per D-07, D-08, D-11 in 03-CONTEXT.md.
name: Weekly menu rebuild

on:
  schedule:
    - cron: '0 6 * * 1'   # Monday 06:00 UTC — fresh for the week (D-11)
  workflow_dispatch:        # Allow manual trigger for smoke-testing the hook

jobs:
  trigger-rebuild:
    runs-on: ubuntu-latest
    steps:
      - name: Trigger Cloudflare Pages deploy hook
        run: curl -X POST "$CF_DEPLOY_HOOK_URL"
        env:
          CF_DEPLOY_HOOK_URL: ${{ secrets.CF_DEPLOY_HOOK_URL }}
```

Key rules enforced by this file (per RESEARCH.md anti-patterns and Pitfall 3):
- The deploy hook URL is NEVER hardcoded — it is read from `${{ secrets.CF_DEPLOY_HOOK_URL }}`
  (GitHub repository secret, added manually by user after CF dashboard setup).
- `curl -X POST` is required (CF deploy hooks reject GET with 405).
- `workflow_dispatch:` enables manual runs for testing without waiting for Monday.
- The job does NOT run `npm run build` or `astro build` — CF does the actual build (D-08).
- Do NOT add a `pull_request:` or `push:` trigger — push builds are handled by CF Git
  integration (D-06), not GitHub Actions.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub repo → Cloudflare Pages | CF build worker clones repo and runs `npm run build`; repo content is trusted |
| GitHub Actions → CF deploy hook | Cron job POSTs to the hook URL; URL is the only credential |
| CF build worker → zz.pizza | Scraper fetches live HTML from third-party site at build time |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | CF deploy hook URL | mitigate | Store URL exclusively in GitHub Secrets (`CF_DEPLOY_HOOK_URL`); never commit to repo; reference via `${{ secrets.CF_DEPLOY_HOOK_URL }}` in workflow — verified by acceptance criteria on Task 2 |
| T-03-02 | Information Disclosure | GitHub Actions run logs | mitigate | `env:` block exposes secret to shell without printing value in logs; `curl` output is benign (CF returns 200 JSON) |
| T-03-03 | Tampering | Supply chain (npm packages) | accept | `cheerio` already pinned to exact `1.2.0` (no caret) per Phase 1 decision T-01-01; no new npm dependencies in this phase |
| T-03-04 | Tampering | Scraper fetch target (zz.pizza) | accept | ZZ Pizza is the trusted data source; if their site is compromised, the worst outcome is bad pizza data or a build failure — low risk, no PII or financial data involved |
| T-03-05 | Denial of Service | Weekly cron failing | accept | If zz.pizza is down at build time, scraper returns 0 items and CF build fails loudly (DATA-03 behavior is correct); v2 DATA-05 covers manual recovery |
</threat_model>
