## Warnings

### WR-01: Fragile `$effect` side-channel for filter-change clearing

**File:** `src/components/PizzaPicker.svelte:17-20`

**Issue:** The effect reads `activeFilter` as a bare expression to register it as a reactive dependency, then sets `result = null`. This is a non-obvious implicit pattern. Svelte 5's compiler tracks reactive reads inside `$effect` callbacks, so it works today, but the intent is invisible to future maintainers ŌĆö it looks like dead code. If the compiler ever hoists or memoises the read differently (plausible in future Svelte versions), the effect silently stops clearing the result.

```svelte
// Current ŌĆö dependency registered via side-channel read
$effect(() => {
  activeFilter; // track filter as a dependency
  result = null;
});
```

**Fix:** Use `$derived` to tie `result` to the filter directly, or make the dependency explicit using a `$watch`-style callback:

```svelte
// Option A: reset in the onclick handler instead ŌĆö no effect needed
function setFilter(f: Filter) {
  activeFilter = f;
  result = null;
}
// Then in markup: onclick={() => setFilter(f.id)}

// Option B: keep effect but be explicit
$effect(() => {
  const _ = activeFilter; // intentional dependency ŌĆö reset result on filter change
  result = null;
});
```

Option A (co-locating the reset with the state change) is the more idiomatic Svelte 5 approach and removes the need for `$effect` entirely for this purpose.

### WR-02: Build fails silently if `prebuild` is skipped

**File:** `src/pages/index.astro:3`

**Issue:** `menu.json` is imported at line 3. The file is generated by `scripts/scrape.js` which runs via the `prebuild` npm hook. If a developer runs `astro build` directly (bypassing `npm run build`) or if a CI environment calls `astro build` explicitly, `menu.json` may not exist and the build will throw a module-not-found error with no clear guidance.

```js
import menuData from '../../public/menu.json';
```

**Fix:** Add a fallback check or document the build entry point clearly. The most robust fix is a guard in `astro.config.mjs` using the `vite` `configResolved` hook, or simply ensuring CI always uses `npm run build`. At minimum, add a comment to the import explaining the dependency:

```js
// Generated by `npm run scrape` (runs automatically via prebuild).
// Do NOT call `astro build` directly ŌĆö use `npm run build`.
import menuData from '../../public/menu.json';
```

This is a documentation fix that prevents real developer confusion with zero code change.

### WR-03: `@astrojs/check` and `typescript` in `dependencies` instead of `devDependencies`

**File:** `package.json:16-24`

**Issue:** `@astrojs/check` and `typescript` are build/type-checking tools with no runtime role. Both are listed in `dependencies`, which means they are installed in production environments. This is incorrect dependency categorisation. The project has no `devDependencies` key at all.

```json
"dependencies": {
  "@astrojs/check": "^0.9.8",
  ...
  "typescript": "^5.9.3"
}
```

**Fix:** Move both to `devDependencies`:

```json
"dependencies": {
  "@astrojs/svelte": "^8.0.5",
  "@tailwindcss/vite": "4.2.2",
  "astro": "^6.1.4",
  "cheerio": "1.2.0",
  "svelte": "^5.55.4",
  "tailwindcss": "4.2.2"
},
"devDependencies": {
  "@astrojs/check": "^0.9.8",
  "typescript": "^5.9.3"
}
```

Note: `astro`, `@tailwindcss/vite`, and `cheerio` can also be argued as devDependencies for a fully static site (all are build-time tools), but the current split at minimum needs `@astrojs/check` and `typescript` moved.

### IN-01: Type assertion on JSON import has no runtime validation

**File:** `src/pages/index.astro:9`

**Issue:** `menuData as MenuData` is a TypeScript-only assertion. If `menu.json` is written by an older or modified version of the scraper that omits a field (e.g. `source_url`), the mismatch is invisible at runtime ŌĆö TypeScript only checks at compile time, and JSON imports bypass strict structural checking via `as`.

```ts
const menu = menuData as MenuData;
```

**Fix:** For this project's risk level this is acceptable (the scraper validates its own output before writing). If schema stability becomes a concern, a lightweight runtime check suffices:

```ts
if (!Array.isArray((menuData as MenuData).pizzas) || (menuData as MenuData).pizzas.length === 0) {
  throw new Error('menu.json is malformed or empty ŌĆö run npm run scrape');
}
const menu = menuData as MenuData;
```

### IN-02: Missing semicolon in `svelte.config.js`

**File:** `svelte.config.js:5`

**Issue:** The export statement is missing a trailing semicolon. This is valid JavaScript in a module context, but inconsistent with the rest of the codebase which uses semicolons (TypeScript files, Astro frontmatter).

export default {
  preprocess: vitePreprocess(),
}   // ŌåÉ no semicolon

export default {
